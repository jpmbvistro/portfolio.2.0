Tic Tac Toe Beans

Made for a coding challenge. Tic Tac Toe Beans was an excellent exercise in diving head first into OOP.....

Made for a coding challenge. Tic Tac Toe Beans was an excellent exercise in diving head first into OOP. The original challenge was to build a two-player tic-tac-toe game that displayed winners on the DOM. Nothing was explicitly tasked to be an object, but I thought it would be a fun to see how far I can quickly abstract a simple board game.

My first idea was to dissect the tiles into its key features as it was the most numerous and therefore had the highest quantitative chance of shortening my code. I then made the boardgame into its own object type. While this wasn't absolutely necessary, I wanted practice reducing the number of global variables by keeping them scoped within objects and having an initializing function 'start' the game. Making the gameboard an abstracted class makes it more portable and allows for the posibility for board sizes beyond 3 x 3

I also abstracted the players into their own class which allowed me to quickly change between text markers on the tiles to images

I also added disabled states with corresponding styling to tiles so that there are visual cues for the user to understand the game state rather than depend on a text prompt.

My biggest challenge was checking for win & tie states. I mistakenly added object dependency from the tiles to the gameboard in an attempt to make win checking more efficient. As a result, checking for a win state created more dependencies between tiles, objects and players. If I were to improve this game, I would better encapsulate the gameboard's role in handling, win/tie states and player turn shifting. While the project was to rapidly prototype the application, I would have found more efficient and readable results if I took a few more minutes to pre-plan object dependencies and roles.

To-Do List

Learning the subtleties of UX/UI design...

This project was made for a coding challenge to create a to-do list. I initially implemented the program with checkboxes which visually struckthrough tasks using a css sibling state change. I was later challenged to implement the program without the checkbox as it better adhered to the separation of concerns by shifting functionality to javascript rather than css.

I decided to try to make the interface more intuitive by allowing users to click tasts to complete them. Since clicking a task can visually suggest editing or completing a task, I added a translucent edit button (which becomes opaque on hover) to suggest to users that editing can only be achieved through the button, which (hopefully) gets users to deduce that clicking the task completes it rather than edits it.

I also added small QoL features such as being able to finish editing the task by clicking away or pressing enter, exiting from editing if the entire tast is backspaced, and automatically enter editing mode when clicking on an empty task (which normally would signal a completion)

Portfolio 2.0

The second iteration of the portfolio you see here...

I wanted to be sure to include the site itself in my portfolio. Custom made, without a template, using HTML, CSS, and Vanilla JS. I wanted to streamline my portfolio from its previous iteration, focusing on my work as a software engineer and including some interactive elements.

The header includes a mouse-over animation that expands my the initials in my handle to their respective names. The animation is a call back to my experiences having to pare down my name as its fairly long. My name already reflects my culture's complex relationship with colonization as each of my names are Hispanic in origin. I decided that this portfolio would be an approppropriate means of sharing my full name without being the focal point of the site.

When choosing how to display my technology stack, I was tempted to display a percent bar indicating my percieved ability/proficiency with each of the technologies. I found this a little confusing and misleading. Is my proficiency with a particular piece of technology measure by my application of it or my comprehension of it? And are these measurements in relation to each other? This suggests I am the least proficient in something, but that visualization might be misconstrued as in competance while in fact I might be very competant with it but just not as so as with my other skills. If you would like to know what I am proficient in, and what I am not, this portfolio should provide a better indication.

My favorite part of building this has been building the masonry layout of my works. I wanted a way of displaying my work that was fun and interactive, showcased my knowledge of complex layouts built with advanced CSS and JS features.

Special credit goes to Jesse Breneman and Andy Barefoot whose articles helped build the hex grid and masonry layout respectively.

Keep up with the News

Keep up with current news and more!

An application made using only HTML, CSS, JS and public APIs. I wanted to test the capabilities of having one API provide information used in another APi. I thought that the news typically gives a nice introduction into what's happening in the world, but I thought it would be even better to get related books to the topic.

I quickly learned that directly passing all related keywords to an API's search query may yield limited results based on how niche and numerous the keywords are.

If I were to reimplement this application, I would parse individual keywords into their own search queries rather than all keywords into one search query.
